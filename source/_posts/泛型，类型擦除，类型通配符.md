### 泛型类和方法

泛型类引入一个类型变量T，用尖括号括起来放在类名后面。类型变量一般使用大写形式：E=element, K=key, V=value, T=type.
<!-- more -->

```java
public class Pair<T>{
	private T first;
	private T second;
	
	public Pair(T first, T second) {this.first = first; this.second = second;}
}
```

泛型方法可以在普通/泛型类中定义，把类型变量放在修饰符后面，返回类型前面。调用时将类型放在函数名前，也可自动推断。

```java
class ArrayAlg{
	public static <T> T getMiddle(T... a){
		return a[a.length/2];
	}
}

ArrayAlg.<String>getMiddle("Alice", "Bob");
ArrayAlg.getMiddle("Alice", "Bob");
```

#### 限定类型变量

使用形如`<T extends BoundingType>`可以限定`T`为`BoundingType`的子类型，`BoundingType`可以是类或接口。可以用`&`连接多个`BoundingType`，但其中只能有一个类，并且类必须是列表中的第一个。类型擦除时会使用第一个限定类型来替换。

### 类型擦除

Java引入泛型时，没有像C#那样选择加入一套平行的泛型化类型，而是为了向后兼容性的考虑采用了类型擦除式泛型实现。泛型只在原码中存在，这种实现效率差于c#的具现化式泛型，但是只需要改动java编译器而无需改变字节码和虚拟机。

裸类型(raw type)视为所有泛型实例的共同父类型，如`ArrayList`是`ArrayList<String>` `ArrayList<Integer>`的裸类型。裸类型删除类型参数，并用第一个限定类型加以替换，如果没有限定类型，用`Object`替换。`Pair<T>`的裸类型是：

```java
public class Pair{
	private Object first;
	private Object second;
	
	public Pair(Object first, Object second) {this.first = first; this.second = second;}
}
```

编译时，所有的泛型类型都会被还原回裸类型，对于元素的访问修改则会自动插入强制类型转换。由于Primitive类型无法转变为`Object`，不支持它们的泛型。

```java
public static void main(String[] args){
	Map<String, String> map = new Hashmap<String, String>();
	map.put("A", "Alice");
	String s = map.get("A");
}

public static void main(String[] args){
	Map map = new Hashmap();
	map.put("A", "Alice");
	String s = (String) map.get("A");
}
```

#### 类型擦除带来的约束

运行时Java虚拟机完全无从得知泛型的任何信息，正因于此以下看似正确的代码是不支持的：

```java
public class TypeErasureGenerics<E>{
    public void dosomething(Object item){
        if(item instanceof E){ //无法对泛型对象进行运行时类型查询
            
        }
        E newitem = new E(); //无法使用泛型创建对象
        E[] itemArray = new E[10]; //无法构造泛型数组
    }
}
```

想要创建对象或者数组可以传入一个Class对象或一个构造器：

```java
public class test {
    public static void main(String[] args) throws Exception {
        TypeErasureGenerics<String> obj = new TypeErasureGenerics<>();
        Class c1 = obj.foo1(String.class).getClass(); //java.lang.String
        Class c2 = obj.foo2(String::new).getClass(); //java.lang.String
    }
}

class TypeErasureGenerics<E> {
    public E foo1(Class<E> t) throws Exception {
        return t.getDeclaredConstructor().newInstance();
    }

    public E foo2(Supplier<E> constr) throws Exception {
        return constr.get();
    }
}
```

此外，不能创建泛型数组：

```
Pair<String>[] arr = new Pair<String>[10]; //错误

Object objarr = arr;
objarr[0] = new Pair<Integer>(); //如果可以创建泛型数组，这一句可以通过编译，但会在运行时导致类型错误。
```

#### Bridge method

```java
interface Comparable <T>{
  public int compareTo(T that);
}
final class NumericValue implements Comparable<NumericValue> {
    private int value;
    public NumericValue(int value) { this.value = value; }
	public int getValue() { return value; }
    public int compareTo(NumericValue that) { return this.value - that.value; }
}
```

类型擦除后会得到：

```java
interface Comparable{
    public int compareTo(Object that);
}
final class NumericValue implements Comparable {
    private int value;
    public NumericValue(int value) { this.value = value; }
	public int getValue() { return value; }
    public int compareTo(NumericValue that) { return this.value - that.value; }
}
```

此时`NumericValue`并没有实现接口中的`compareTo`方法，类型擦除前相同签名的两个函数，现在变得签名不同了。

为了解决这个问题，保证`NumericValue`仍然正确实现了`Comparable`接口，编译器会合成一个桥方法：

```java
public int compareTo(Object that) { return this.compareTo((NumericValue)that);}
```

桥方法仅仅是为了保持多态正确性而存在，并不代表就可以用一个任意类型的对象调用`NumericValue.compareTo(Object)`方法：

```java
n.compareTo(n); //ok, 0
n.compareTo("Alice"); //error
```

如果使用反射强行调用`compareTo(Object)`，会在类型转换的时候发生一个`ClassCastException`.

```java
NumericValue n = new NumericValue(2);
Method m = n.getClass().getDeclaredMethod("compareTo", Object.class);
m.invoke(n, "Alice"); //ClassCastException
```

### 通配符类型

对于泛型类而言，无论类`S`和类`T`有什么联系，`Pair<S>`和`Pair<T>`都没有任何联系。泛型类可以扩展/实现另一个泛型类，比如`ArrayList<T>`实现了`List<T>`接口，也就是说`ArrayList<Integer>`是`List<Integer>`的子类。但`ArrayList<Integer>`与`ArrayList<Number>`没有关系。

在通配符类型中，允许类型参数的变化，比如一个`List<? extends Number>`类型的变量允许被赋值为`ArrayList<Number>`,`ArrayList<Integer>`对象；一个`List<? super Integer>`类型的变量允许被赋值为`ArrayList<Integer>`,`ArrayList<Number>`对象。

#### PECS原则

在使用通配符类型时，要注意`? extends Classname`规定了类型的上限：对于`List<? extends Number>`的变量`foo`，编译器无从得知`foo`究竟指向一个`List<Number>`,`List<Integer>`还是`List<Double>`，这三种情况都是合法的。这使得我们无法向foo中增加任何一个类的对象：

    1. 增添一个`Number`超类的对象：无论`foo`指向存储什么类型对象的`List`都不合法

 	2. 增添一个`Number`对象（这里不考虑`Number`其实是抽象类）：如果`foo`是`List<Integer>`或`List<Double>`，不合法
 	3. 增添一个`Integer`对象：如果`foo`是`List<Double>`，不合法

等等。总可以找到一个`foo`的可能类型，使得增加元素操作不合法，因此编译器不能允许这种操作通过编译。然而，从`foo`中读出一个值是完全可以的。`foo`中存储的元素一定是某个`Number`及其子类的元素，可以用一个`Number`类型对象来接收这个值。

相反，`? super Classname`规定了类型的下限。对于`List<? super Integer>`的变量`bar`，编译器允许向`bar`中添加任何一个`Integer`子类的对象；但若想从`bar`中读取值，必须要用`Object`类型变量接收。

上文所述的添加对象，泛指一切写入泛型对象的`set`方法；读取值，泛指一切读取泛型对象的`get`方法。

由此得到所谓的PECS原则：Producer Extends Consumer Super. 这是从泛型对象的角度来谈的：如果我们只从一个泛型对象（如，列表）中获取对象，它就是一个Producer，可以使用`extends`. 反之如果希望写入泛型对象，它就是Consumer，应该使用`super`. 如果同时需要读取和写入，则`extends` `super`都不应该使用。
